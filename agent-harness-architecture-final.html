01 背景：什么是 harness

在动手写任何代码前，先把概念固定下来——这是工程师做出取舍的前提。

「Agent」这个词被滥用得很厉害。市面上大量 no-code 工作流、prompt 编排库，把「拖几个节点把 LLM 调用串起来」也叫作 agent。但凡接触过强化学习或者真正的智能体训练，就知道这套东西和 agent 没什么关系——它只是带 LLM 节点的脚本。

真正的 agent 由两件东西构成：

  • 智能体（intelligence）：一个被训练出来、能感知-推理-行动的神经网络模型。DQN、AlphaStar、OpenAI Five、Claude，本质都是这样一个模型。
  • 载体（harness）：这个模型运行所在的环境——能让它感知输入、调用工具、影响世界的所有代码与基础设施。
Agency comes from training. Harness comes from engineering.
模型决定走哪条路，harness 决定路是什么材质修的。

本设计文档的目的，是给「harness」这件事一份完整且可落地的工程蓝图。模型已经由 Anthropic 训练好；要做的是把它装进一辆能跑、能停、能加油、能避险的车里。

harness 工程师的真正职责

  • 实现工具：给 agent 装上手——文件读写、shell、API 调用、浏览器控制。
  • 策划知识：给 agent 装上图书馆——文档、API spec、领域知识，按需加载。
  • 管理上下文：给 agent 装上记忆系统——子代理隔离、上下文压缩、跨会话持久化。
  • 控制权限：给 agent 装上刹车——沙盒、审批、信任边界。
  • 采集轨迹：给 agent 装上行车记录仪——每一次行动都是下一代模型的训练信号。

02 第一性原理：Agent = Model + Harness

一个 agent 产品，永远不止是模型，也不止是代码。它是「模型 + 这个模型所栖居的世界」。

Agent Product = Model + Harness
Tools Knowledge Observation Action Interface Permissions

模型决定，harness 执行。模型推理，harness 提供上下文。模型是司机，harness 是车。

这条等式贯穿全文。后面的所有讨论——8 个关注面、模块依赖、6 阶段实施——都是为了把右边的 harness 具体化成一份可落地的代码组织。

03 核心循环：harness 的最小内核

每一个机制都是叠在同一个循环上的。这个循环永远不变；机制是外圈的累加。

→
Step 2

→
Step 3
Tool Execute
bash · read · write · edit ·…
→
Step 4
tool_result
append to messages[]
stop_reason == "tool_use" 继续循环   ·   否则 return 
def agent_loop(messages):
    while True:
        response = client.messages.create(model=MODEL, system=SYSTEM,
                                          messages=messages, tools=TOOLS)
        messages.append({"role": "assistant", "content": response.content})
        if response.stop_reason != "tool_use":
            return
        results = [run_tool(b) for b in response.content if b.type == "tool_use"]
        messages.append({"role": "user", "content": results})
# 所有机制都叠在这个循环之上 -- 而不动循环本身。

04 按关注面组织的模块架构

harness 不是自下而上的层级栈——把模块按 L0–L9 编号会制造一种不存在的「上层优于下层」的错觉。 真实的依赖关系是网状的：同一个模块可能同时被核心循环、安全面、观测面依赖；有些模块（如观测）横切所有面。 以下按关注面（plane）组织 22 个模块，同一面内的模块职责相近，面与面之间没有层级高低。

05 模块依赖关系

上一节的「面」告诉我们模块的职责归属。但真实依赖不是「A 依赖 B」这么干净——一个模块可能被多个面的模块同时依赖。 下面列出关键依赖边，按模块逐一标注。

依赖总表

core/modelArt/config, obs/cost, ctx/cache API 客户端——cost 跟踪和 cache 标记是内嵌的，不是外挂
core/messagesArt/logging 消息追加即事实，日志是其持久化影子

关键依赖特征

  • rt/config 是被依赖最多的模块：几乎所有模块都需要读配置。它是唯一没有依赖的模块之一。
  • 安全面是拦截器，不是层：tools/dispatcher → safe/permission + safe/hooks 是调用链上的必经路径，但安全面不在工具「之上」或「之下」——它在管线中间。
  • core/messages 是唯一可变状态：loop 读它、model 消费它、transcript 记录它、compact 重写它——messages 是整个 harness 的状态中枢。
  • 观测面并不「在顶端」：obs/cost 被 core/model 内嵌调用，obs/transcript 被 core/messages 和 ctx/compact 双向依赖。观测不是事后附加，而是执行过程中织入的。
  • 环依赖规避：core/model → obs/cost（记录 token），但 obs/cost 不回调 core/model。同样，ctx/compact → core/messages（压缩写入），但 messages 不主动调 compact。所有跨面依赖都是单向的。

06 6 周实施计划

6 个阶段，每个阶段产出可运行的增量。不是 waterfall——每周都有可跑的最小 harness。

让 agent 能跑完一个完整的 tool_use 循环——哪怕只有 3 个工具。

08 结语：先建好车，模型自然会开

这份文档把「为什么」「是什么」「怎么做」串成了一条线。从第一性原理出发——智能由训练而来，agent 产品 = 模型 + harness——再到最小循环、按关注面组织的 8 面 22 模块、网状依赖关系，最后落到 6 周的具体节奏。

不用层级（L0–L9）来组织架构，是因为 harness 的本质不是栈。安全面拦截工具调用，但它不在工具「之上」——它在管线中间。观测面记录 token 消耗，但它不在系统「顶端」——cost tracker 被 model client 内嵌调用。上下文面注入 memory 和 skill，但它和持久面的 session 管理是并列关系。用「面」替代「层」，是为了让架构图说真话。

别去想着「重新发明 agent」。要做的是一台车——把 Anthropic 训练出来的智能装进去，让它能感知、能动手、能避险、能复盘。每一个面都对应一类外部世界的需求；每一个模块都为模型解决一个具体的「我看不到」或「我做不到」。
